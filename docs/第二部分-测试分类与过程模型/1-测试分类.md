# 测试分类（多维分类体系）

### 按开发阶段划分（测试级别）

1. **单元测试（Unit Testing）**：针对程序模块（函数、类）的测试
2. **集成测试（Integration Testing）**：测试模块之间的接口和交互
3. **系统测试（System Testing）**：测试完整的系统功能
4. **验收测试（Acceptance Testing）**：用户验收，确认系统是否符合需求

### 按测试实施组织划分（必考点）

这是本节课最重要的理论考点，必须区分清楚。

| 分类 | 别称 | 实施者 | 环境 | 核心目的/思维 |
| :--- | :--- | :--- | :--- | :--- |
| **开发方测试** | 验证测试 / α测试 | 开发人员 (Developer) | 开发环境 | **正向思维**：证明软件实现了规定功能（Verification - "Are we building the product right?"）。 |
| **用户测试** | 确认测试 / β测试 | 最终用户 (User) | 实际使用环境 | **确认/评价**：核实软件是否符合用户预期（Validation - "Are we building the right product?"），关注适用性。 |
| **第三方测试** | 独立测试 | **独立机构** | 模拟/真实环境 | **逆向思维**：尽可能多地发现潜在缺陷。保持客观、公正、权威。 |

#### 第三方测试详解（重点概念）

**定义**：在**技术、管理、财务**上与开发方（乙方）和用户方（甲方）完全独立的组织进行的测试。

**原则**：
- 独立公正原则（核心）
- 委托原则（甲方或乙方委托均可）
- 回避原则（利益相关需回避）
- 保密原则

**必要性**：
- 打破开发人员的"定势思维"（自己写的代码看不出毛病）
- 具备专业的测试工具和经验（复用性高，成本摊薄）
- 作为合同验收的客观依据（如国家重点研发计划验收，必须有第三方报告）

**常见类型**：确认测试、鉴定测试、验收测试

**【考试重点】**：
- 一定要能区分**开发方测试**、**用户测试**和**第三方测试**
- 考试可能会给出一个场景，问你这属于哪种测试？或者问"由中立机构进行的测试叫什么？"
- **关键词记忆**：第一方（开发自测）、第二方（用户测）、第三方（独立机构测）

### 按测试方法划分

1. **黑盒测试（Black-Box Testing）**：基于规格说明书
2. **白盒测试（White-Box Testing）**：基于源代码或详细设计
3. **灰盒测试（Gray-Box Testing）**：结合黑盒和白盒

**注意**：这些分类基于不同的标准，不应混淆。

### 测试分类的三维空间（The "Cube"）

测试分类是一个三维空间，**考试时注意分类标准不要混淆**：

1. **按阶段/层次 (Phase/Level)**：
   - 单元测试 (Unit) -> 集成测试 (Integration) -> 系统测试 (System) -> 验收测试 (Acceptance)

2. **按方法/可见性 (Method)**：
   - 白盒 (White-box): 看得见代码
   - 黑盒 (Black-box): 看不见代码，基于功能
   - 灰盒 (Grey-box)
   - **动/静态**：静态（文档检查、代码走查）、动态（运行程序）

3. **按目标/特性 (Target/Characteristic)**：
   - 功能、性能、安全性、兼容性、可靠性等

**【考试重点】**：
- 会给出一堆测试名词，问分类标准是否统一
- 例如：不能把"白盒测试"和"单元测试"混为一谈，一个是方法，一个是阶段
