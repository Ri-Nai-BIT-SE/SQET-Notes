### 第一部分：课堂笔记 (Lecture Notes)

本章内容主要围绕“实操”和“项目驱动”，旨在指导学生如何将测试理论应用于实际的生产实践和课程实验中。

#### 1. 组建测试队伍 (Building the Test Team)

**1.1 测试团队的地位与责任**
*   **核心理念：** 测试不仅是找错，更是质量保证（QA）体系的一部分。
*   **基本职责：**
    *   发现问题（正向/逆向思维结合，主流观点倾向于Myers的逆向思维：测试是为了发现错误）。
    *   督促开发修复缺陷（Defect Fix）。
    *   **重点流程：** 发现缺陷 -> 开发修复 -> 测试人员确认（Verify/Regression）-> 关闭缺陷（Close）。
    *   **额外职责：** 帮助制定开发计划（作为辅助）、改善流程、提高代码规范性（易读性）。
*   **组织模型（三种）：**
    1.  **以开发为核心：** 测试人员归开发经理管。
    2.  **以项目经理为核心：** 测试组长与开发组长平级，都汇报给PM。
    3.  **三国鼎立：** 质量/测试经理、开发经理、项目经理并列（大企业常见，有独立话语权）。
*   **团队规模：**
    *   **估算：** 依据测试范围、功能点、复审时间等进行科学估算（类似《人月神话》），而非拍脑袋。
    *   **Buffer：** 长期团队需要预留10%-20%的富裕量（应对突发任务、人员调配、多项目并行）。
    *   **人员配比：** 操作系统类（2:1，测试要求最高）、应用平台类（1:1）、特定应用类（1:2~1:4）。

**1.2 测试团队的构成与分工**
*   **角色细分：**
    *   **测试经理/组长：** 负责统筹、计划制定（Plan）、协调资源。**强调：测试计划应由组长制定。**
    *   **配置管理员（重要但常被忽视）：** 管理代码、文档（测试计划、用例）、版本构建。需考虑文档与代码的版本对应关系。
    *   **实验室管理员：** 负责硬件、网络、环境搭建及权限管理。
    *   **测试设计人员 vs. 测试执行人员：** 课程强调二者分离。资深人员做设计（Design），初级人员做执行（Execution）。
    *   **内审员：** 审查流程、模板建立、报告质量。
*   **职业发展：** 初级（执行、发现问题） -> 中级（工具使用、脚本开发） -> 资深（架构设计、策略制定、评审）。
*   **微软测试工程师的一天（案例）：**
    *   **自动化：** 每日构建（Daily Build） -> BVT（基本验证测试）。
    *   **流程：** 上班先看测试报告 -> 处理最高级别Bug（Pri 0，当日修） -> 验证已修复Bug -> 回归测试（Regression） -> 开发新脚本。

**1.3 管理与激励**
*   **激励：** 明确奖惩，支持合理工作方式。
*   **培训：** 知识共享，在岗培训（Senior带Junior）。

#### 2. 测试需求分析与测试计划 (Analysis & Planning)

**2.1 测试目标与准则**
*   **测试目标：**
    *   不是为了填文档，而是要有明确目的（例如：评估旧版本质量、验证新特性、回归测试）。
    *   **关键指标：** 测试执行率、执行通过率、严重缺陷修复率、缺陷解决率。
*   **出入准则（常被忽视）：**
    *   **进入准则 (Entry Criteria)：** 什么时候可以开始测？（代码是否冻结？文档是否齐备？环境是否Ready？）
    *   **退出准则 (Exit Criteria)：** 什么时候测完？（不仅是时间到了，而是质量目标达成了）。

**2.2 测试需求分析**
*   **过程：** 了解背景 -> 分析业务/用户需求 -> 确定功能/非功能特性 -> 优先级划分。
*   **功能与非功能：**
    *   **功能性：** 覆盖率。
    *   **非功能性：** 性能、安全性。**注意：** 做性能测试要有理由（基于瓶颈分析），而不是为了得分硬凑。
*   **优先级：** 资源有限时，先测核心功能。

**2.3 估算与进度**
*   **WBS (工作分解结构)：** 任务层层分解，粒度越细，估算越准。
*   **风险管理：**
    *   必须结合实际（例如：识别“考试周”导致人员无法投入工作的风险）。
    *   要有应对措施（Mitigation）。

**2.4 测试计划的内容 (基于GB/T 9386-2008)**
*   **文档要素：** 标识符（ID）、引言、测试项、测试策略、通过/失败准则、任务分配、环境、风险、**批准（签字）**。
*   **动静结合：** 计划中应包含静态测试（文档评审）和动态测试。
*   **策略选择：** 黑盒/白盒，手工/自动化，根据项目特性（如航天项目与学生作业质量要求不同）选择不同策略。

### 第二部分：考试重点 (Exam Focus)

根据老师在课堂上的明确提示（“这是考点”、“这里会出题”），请重点复习以下内容：

#### 1. 题型预测
*   **单项选择题：** 考察概念的综合运用。
    *   结合第一章和本章内容，考察测试分类、缺陷定义、英文表述。
    *   考察**人员职责**：例如，“以下哪个任务属于测试经理的职责？”或“谁负责关闭缺陷？”
    *   考察**观点判断**：例如，“测试越早介入费用越低”等观点的正误。
*   **案例分析题（偏文科/管理）：**
    *   **情景纠错：** 给出一段项目管理的描述，让你找出其中存在的问题（例如：人员配置不当、流程缺失、缺乏配置管理、没有进入准则等）。
    *   **质量分析报告：** 基于图表（如缺陷趋势图）分析产品质量，判断是否可以发布（Release）。

#### 2. 核心考点 (Key Concepts)

1.  **缺陷生命周期与职责分配 (高频考点)**
    *   **谁发现谁确认：** 发现Bug的人负责验证Bug是否修复。
    *   **关闭Bug (Close)：** 只有**测试人员**有权关闭Bug，而不是开发人员。开发人员只能标记为“已修复（Fixed/Resolved）”。
    *   **流程：** Open -> Fixed -> Verified -> Closed。

2.  **测试团队组织结构**
    *   理解三种模型（开发核心、PM核心、三国鼎立）的区别和优缺点。
    *   不同应用类型下开发与测试的人员比例（OS类 2:1，应用平台 1:1，特定应用 1:2~1:4）。

3.  **测试计划与策略**
    *   **WBS**：如何进行任务分解。
    *   **风险管理**：必须包含风险识别和应对计划（不仅仅是列出风险）。
    *   **GB/T 9386标准**：熟悉测试计划文档的基本结构（标识符、测试项、通过准则、批准等）。

4.  **测试类型与阶段**
    *   **静态 vs 动态**：评审（Review/Inspection）属于静态测试，运行代码属于动态测试。
    *   **测试设计与执行分离**：理解为什么要分离，以及各岗位的技能要求。

5.  **实际应用中的细节 (老师反复强调的扣分点)**
    *   **配置管理**：团队中必须有人负责文档和代码的版本管理。
    *   **标准依据**：做计划和报告时要有标准支撑（国标、行标）。
    *   **目标与准则**：要有明确的进入和退出准则，不能仅仅因为时间到了就停止测试。
