## 📝 第一部分：学生测试工具分享
*(注：这部分内容主要涉及扩展知识，了解工具类型即可)*

### 1. CUnit (针对 C 语言)

*   **类型：** 轻量级单元测试框架，xUnit 架构。
*   **特点：** 跨平台（Linux/Windows），提供断言检查结果，管理测试套件。
*   **架构：** 注册中心 -> 测试套件 -> 测试用例。
*   **优缺点：** 编译快、易集成；但配置繁琐，缺乏 Mock 支持。

### 2. Web 前端静态分析工具

*   **ESLint / StyleLint / HTMLHint：**
*   **目的：** 不运行代码，进行静态扫描。
*   **作用：** 检查语法错误、统一代码风格（如变量命名、字体定义、标签嵌套）。
*   **流程：** 安装依赖 -> 配置规则 -> 扫描 -> 修复报告。

### 3. Google Test (gtest) (针对 C++)

*   **类型：** C++ 单元测试框架。
*   **核心特性：**
    *   **断言：** `EXPECT_` (非致命，失败继续) vs `ASSERT_` (致命，失败终止)。
    *   **Test Fixtures (固件)：** `TEST_F`，用于多个测试用例共享资源/初始化（SetUp/TearDown）。
    *   **死亡测试 (Death Tests)：** 验证程序是否按预期崩溃（如检测内存溢出）。
*   **应用：** 支持自动化回归测试。

## 📝 第二部分：测试应用技术 (重点：性能与本地化)
*(对应 PPT 第5章 & 录音讲解)*

### 1. 负载压力测试（Performance Testing）—— **案例分析题重点**

#### 考查形式

案例分析题（13分中的一部分）。通常给出一个场景描述和**测试工具生成的图表**，要求分析瓶颈。

#### 测试目的

1. 检测系统性能是否满足 SRS 中的非功能性需求（如响应时间、并发数）。
2. 预见系统的负载承受力（峰值、边界）。
3. **发现系统瓶颈并进行调优**（最难点）。

#### 关键指标与分析经验（必背）

**交易响应时间过长**：
*   可能是数据库操作耗时
*   *调优：* 建立索引、优化 SQL、水平分割大表

**CPU 占用率（持续 > 95%）**：
*   瓶颈在 CPU
*   *调优：* 增加处理器、更换更快 CPU、优化算法
*   **合理范围是 60%-70%**

**内存/缓存点击率（持续 < 80%）**：
*   瓶颈在内存
*   *调优：* 增加内存

**分段分析**：
*   通过工具分别模拟访问 Web 服务器、应用服务器、数据库
*   通过减法确定慢在哪一段

### 2. 软件本地化测试 (Localization - L10N) —— **选择/判断题考点**

#### 核心概念

*   **国际化 (I18N - Internationalization)**：
    *   是基础
    *   设计时使代码能处理多语言（如支持 Unicode、分离资源文件）
    *   属于 Engineering 阶段
*   **本地化 (L10N - Localization)**：
    *   是过程
    *   将产品加工以满足特定市场（语言、文化）
*   **关系**：国际化是本地化的前提

#### 测试重点

**不仅仅是翻译**：
*   还包括度量衡（英制/公制）
*   时区
*   货币格式
*   键盘布局

**文化传统**：
*   尤其是**文本方向**（如阿拉伯语从右向左）
*   宗教禁忌
*   颜色喜好

**文本扩展 (Text Expansion)**：
*   翻译后文本变长导致 UI 溢出或截断
*   如英文转德文通常变长

**功能性**：
*   本地化后不能破坏原有功能
*   如快捷键冲突、热键设置

## 📝 第三部分：自动化测试 (Test Automation)
*(对应 PPT 第6章，老师未细讲，但有考点)*

### 考查形式

*   **选择题最后一道（约1分）** + 可能涉及案例分析的背景

### 核心观点（判断正误的关键）

**误区（判断题考点）**：

1. ❌ **"自动化测试能完全取代人工测试"** → **错误**
   *   自动化测试主要用于回归测试、负载压力测试
   *   人工测试在探索性测试、易用性测试等方面不可替代

2. ❌ **"自动化工具能发现大量新缺陷"** → **错误**
   *   自动化主要是回归测试，保底作用
   *   主要用于验证已有功能是否被破坏

3. ❌ **"任何场景都适合自动化"** → **错误**
   *   不稳定、一次性、需要物理交互的项目不适合

**正确观点**：

*   自动化测试主要用于**回归测试**、负载压力测试、以及大量重复的执行工作
*   **录制/回放**是早期的自动化技术
*   现在的趋势是**数据驱动**和**关键字驱动**
