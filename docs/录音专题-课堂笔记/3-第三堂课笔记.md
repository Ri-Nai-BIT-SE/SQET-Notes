### 二、 小组汇报点评与经验总结 (实践部分)

老师点评了第1、5、7组的测试计划汇报，指出了共性问题和加分项：

1.  **关于“小学期项目”的二轮测试（核心痛点）：**
    *   **问题：** 很多组测的是大二/大三小学期的旧项目。老师担心大家直接拿当时的测试文档来糊弄这门课。
    *   **正确做法：** 必须说明“当前状态”。
        *   如果当时有Bug没修：说明本轮测试是**修复+回归测试**。
        *   如果当时已测完：说明本轮是**复用旧用例+新设计用例**，或者进行了**新功能迭代（New Feature）**。
        *   **迭代测试策略：** 如果有新功能，针对新功能做单元/集成/系统测试；针对旧功能做回归测试。
2.  **测试计划与管理：**
    *   **进度图：** 不要只列时间段（如“3天”），要画**甘特图**（Gantt Chart），体现任务的并行、串行关系和依赖关系。
    *   **人员分工：** 不要只写人数，要**具体到人名**。老师打分和提问是针对个人的。
    *   **风险管理：** 不要只写“时间不够”，要预留Buffer（缓冲期）。
3.  **加分点（参考第7组）：**
    *   定义了**缺陷严重程度标准**（Severity Levels），而不是凭感觉定Bug等级。
    *   明确了测试的切入点（如：开发方测试 vs 乙方测试）。
    *   涵盖了基本的**功能、易用性、文档检查**，并额外增加了**性能/兼容性**等专项测试。

### 三、 理论课笔记：软件测试基础

#### 1. 软件测试职业发展 (PDF P3-9)
*   **必要性：** 未经测试的软件不具备可靠性。测试成本通常占开发总成本的50%甚至更多。
*   **职业定位：**
    *   测试不是开发的“垫脚石”，而是一个专业技术学科。
    *   职业路径：测试工程师 -> 测试组长 -> 质量保证经理 -> 技术总监/VP。
*   **认证体系：**
    *   **国内：** 软考（软件评测师），属于中级职称（以考代评），由人社部和信产部颁布。
    *   **国际：** ISTQB（国际软件测试认证委员会），分为初级(CTFL)、高级(CTAL)、专家级(CTEL)。
    *   **对比：** 找工作或企业招投标时，不同认证有不同含金量。

#### 2. 测试组规模与配比 (PDF P10 - **考点预警**)
*   引用了 **E.Dustin** 等人的建议表。
*   **核心逻辑：** 开发人员与测试人员的比例不是固定的，取决于产品类型、市场规模、风险等级。
    *   商业软件（大市场）：开发:测试 ≈ 3:2 或 接近 1:1（测试占比高）。
    *   企业内部系统：比例可能较低（如 4:1）。
*   **注意：** 老师提到以前出过相关**案例分析题**，考察为什么项目出问题（原因可能是测试人员配比不当）。

#### 3. 典型软件失效事件 (案例研究) (PDF P11-23 - **考试重点**)
老师强调：以后考试的大题（30分主观题）通常是**一黑（黑盒）、一白（白盒）、一案例**。案例题常出自这里。

*   **Therac-25 放射治疗仪 (1985-1987)**
    *   **后果：** 6人治疗过量，3人死亡。
    *   **原因：**
        1.  系统安全性分析未包含软件（只信任硬件连锁，软件去掉了硬件保护）。
        2.  **重用代码问题**：重用了T-20的软件代码，但硬件环境变了（假设和前提改变），未重新测试。
*   **Ariane 5 (阿里安5号) 火箭爆炸 (1996)**
    *   **后果：** 发射40秒后爆炸。
    *   **直接原因（技术）：** 数据类型转换溢出。将 **64位浮点值** 转换为 **16位有符号整数** 时溢出。
    *   **根本原因（管理/测试）：**
        1.  直接重用了Ariane 4的代码。
        2.  Ariane 5的水平速度比Ariane 4快得多，超出了原代码的适用范围。
        3.  **在新的环境下完全没有进行测试**。
*   **火星探测器 (Mars Climate Orbiter) (1999)**
    *   **后果：** 坠毁/失联。
    *   **原因：** **度量衡单位不统一**。地面软件使用**英制**（磅力），飞行器软件使用**公制**（牛顿）。
    *   **教训：** 多团队协作（不同分包商）时的接口测试和标准统一至关重要。
*   **实际案例（近期/国内）：**
    *   **国家金库系统：** 字符写错导致全国报表错误。
    *   **12306：** 早期崩溃、扣钱不出票（高并发问题）。
    *   **WTT 2025中国大满贯购票系统：** 实名认证异常（数据同步延迟）、场馆信息错误（八极场vs二号馆）。
    *   **工商银行App (2025年10月29日)：** 出现资产显示异常（余额归零或乱码），属于极端的软件故障案例。

### 四、 考试重点总结 (Exam Focus)

根据老师录音中的暗示和强调，以下内容极大概率出现在考试中：

1.  **案例分析题 (Case Study)：**
    *   **核心考点：** 给你一个失败的场景，问你原因是什么？违反了什么测试原则？
    *   **必背案例细节：**
        *   **Ariane 5** = **浮点转整数溢出** + **代码重用未测试**。
        *   **火星探测器** = **公制与英制单位混用**。
        *   **Therac-25** = **软件未包含在安全性分析中** + **重用代码前提改变**。
2.  **测试团队配置：**
    *   理解测试人员配比的逻辑（E.Dustin的建议）。如果案例中提到“开发20人，测试1人，结果项目上线后Bug频出”，原因就是**测试投入不足/配比失调**。
3.  **测试流程与术语：**
    *   理解 **单元测试 -> 集成测试 -> 系统测试 -> 回归测试** 的顺序和含义。
    *   特别是在已有项目（如Ariane 4到5，或学生的小学期项目）中，**回归测试 (Regression Testing)** 的重要性。
4.  **黑盒/白盒测试：**
    *   虽然本节课讲得少，但老师明确说了期末有“一黑一白”大题，后续课程会详细讲方法（等价类、边界值、路径覆盖等）。
