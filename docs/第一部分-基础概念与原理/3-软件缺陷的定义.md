# 软件缺陷（Bug）的定义

### 缺陷的定义（5条规则，重点）

软件缺陷是指软件产品与预期属性之间的偏差。一个缺陷存在于以下情况：

1. **未达到产品说明书标明的功能**
2. **出现了说明书指明不会出现的错误**
3. **功能超出了说明书范围**（画蛇添足）
4. **未达到虽未指出但应达到的目标（隐含需求）**
5. **测试人员/用户认为不好**（难理解、难用、慢）

**【录音重点】**：第5点体现了以客户为中心，即使功能对了但体验极差也是Bug。

### "Bug"一词的历史来源

"Bug"（虫子）一词来源于早期计算机时代：一只飞蛾飞进了计算机，导致系统故障。从此，"bug"成为软件缺陷的代名词。

### 缺陷产生的原因（考点）

**最大来源**：**产品规格说明书 (Specification)** 是缺陷的最大来源。

**【录音重点】**：很多人误以为Bug主要来自代码（Code），其实大部分也是修复成本最高的Bug来自于需求和设计阶段的文档错误。

**缺陷引入的阶段**：
- 需求分析阶段：需求理解错误、需求遗漏、需求不一致
- 设计阶段：设计错误、设计遗漏
- 编码阶段：编码错误、逻辑错误
- 测试阶段：测试用例设计错误、测试环境配置错误

**缺陷传播**：从一步到下一步的过程中会引入新的缺陷，包括沟通中的误解。

**黑盒测试的局限性**：
- 如果需求规格说明书中规定了多余的功能或遗漏了某些功能，黑盒测试无法查出
- 需求规格说明书本身也可能不完全正确

### 修复成本

随着生命周期推移（从需求到发布），修复成本呈**几何级数**增长。
