# 缺陷管理（Defect Management）

这部分考点非常细，容易在流程上挖坑。

### 🐛 缺陷生命周期 (Bug Life Cycle) —— **核心考点**

请背下这个标准流程：

**基本流程**：发现 (New/Open) -> 修复 (Fix) -> 验证 (Verify) -> 关闭 (Close)

**详细流程**：

1.  **New/Open (新建/打开)**：测试人员发现Bug，提交给开发。
2.  **Assigned (已分配)**：项目经理或开发组长将Bug分配给具体开发人员。
3.  **Fixed/Resolved (已修复/已解决)**：开发人员修改代码后，将状态改为已修复。
4.  **Verified (已验证/回归测试)**：**测试人员**去验证这个Bug是否真的修好了。
5.  **Closed (已关闭)**：验证通过，**测试人员**关闭Bug。

**状态管理**：Open, Fixed, Closed, Deferred (延期), Rejected (拒绝), Re-open (重开)

**⚠️ 必考陷阱：**
*   **谁来关闭Bug？** 只有**测试人员**！开发人员修好了只能标"Fixed"，绝对不能自己关。
*   **回归测试 (Regression Testing)**：修好一个Bug后，不仅要测这个Bug，还要测相关联的功能，防止"引入了新的Bug"。

**协同工作**：它是测试与开发协同的过程，**不仅仅是测试人员的事**。

**角色职责**：
- **测试人员**：发现、报告、复现、验证、关闭
- **开发人员**：确认、修复、调试
- **项目经理 (PM)**：仲裁争议、分配缺陷

**缺陷报告质量**：
- **一图胜千言**：必须有截图
- **可复现性**：描述清晰，步骤明确。如果无法复现，可能会被搁置

**严重性 (Severity) vs 优先级 (Priority)**：
- 严重性由Bug本身决定（如系统崩溃）
- 优先级由项目进度和业务决定（如紧急修复）
- *注：* 并非所有严重Bug都必须立即修复（视发布策略而定）

### 📈 缺陷指标（选择题/计算题）

*   **缺陷密度** = 缺陷数量 / 代码行数（或功能点）
*   **缺陷清除率 (Defect Removal Efficiency)**：考题可能会给场景让你计算
*   **测试用例通过率** = 通过数 / 总执行数
*   **覆盖率**：需求覆盖率、代码覆盖率
*   **信心指数**：用于质量评估

**趋势分析**：
- **S曲线**：进度控制
- **NOB曲线 (Number of Open Bugs)**：判断系统是否趋于稳定，是否可以发布

**质量评估**：最终能不能上线，需要给出一个确定的结论（通过/不通过/优秀/良好），且要有数据支撑（信心指数）

### 流程说明

```
发现(Open) → 分配(Assigned) → 修复(Fixed) → 验证(Verified) → 关闭(Closed)
```
