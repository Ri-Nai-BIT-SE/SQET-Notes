# 自动化测试 (Test Automation)

## 核心观点（判断正误的关键）

### ❌ 常见误区（判断题考点）

#### 误区1：自动化测试能完全取代人工测试

**错误！**

**正确观点**：
*   自动化测试主要用于**回归测试**、负载压力测试、以及大量重复的执行工作
*   人工测试在以下方面不可替代：
    *   探索性测试
    *   易用性测试（主观性）
    *   需要创造性思维的测试
    *   一次性测试场景

#### 误区2：自动化工具能发现大量新缺陷

**错误！**

**正确观点**：
*   自动化主要是回归测试，**保底作用**
*   主要用于验证已有功能是否被破坏
*   自动化测试通常不会发现新缺陷，而是确保已有功能正常工作

#### 误区3：任何场景都适合自动化

**错误！**

**不适合自动化的场景**：
*   **不稳定的功能**：频繁变更的功能
*   **一次性测试**：只执行一次的测试
*   **需要物理交互的项目**：如硬件测试、触摸屏测试
*   **易用性测试**：主观性强的测试

### ✅ 正确观点

*   自动化测试主要用于：
    *   **回归测试**（主要用途）
    *   **负载压力测试**
    *   **大量重复的执行工作**
*   **录制/回放**是早期的自动化技术
*   现在的趋势是**数据驱动**和**关键字驱动**

## 自动化测试工具分类

### 1. 单元测试框架

*   **CUnit**（C语言）：轻量级，xUnit架构
*   **Google Test (gtest)**（C++）：支持断言、Test Fixtures、死亡测试
*   **JUnit**（Java）
*   **pytest**（Python）

### 2. 静态分析工具

*   **ESLint / StyleLint / HTMLHint**（Web前端）：
    *   不运行代码，进行静态扫描
    *   检查语法错误、统一代码风格
    *   流程：安装依赖 -> 配置规则 -> 扫描 -> 修复报告

### 3. 功能测试工具

*   **Selenium**（Web自动化）
*   **Appium**（移动端自动化）
*   **QTP/UFT**（商业工具）

### 4. 性能测试工具

*   **LoadRunner**
*   **JMeter**
*   **Apache Bench**

## 自动化测试的优势与局限

### 优势

*   **提高效率**：快速执行大量测试用例
*   **提高准确性**：减少人为错误
*   **支持回归测试**：确保新功能不破坏旧功能
*   **可重复执行**：24/7运行

### 局限

*   **不能完全取代人工测试**
*   **主要用于回归测试**，不是发现新缺陷
*   **需要维护成本**：代码变更需要更新测试脚本
*   **不适合所有场景**：不稳定、一次性、需要物理交互的项目

3. ❌ "任何场景都适合自动化" → **错误**

*   自动化测试的主要用途（回归测试）
*   不适合自动化的场景（不稳定、一次性、物理交互）
*   自动化测试工具的分类
