## 一、判断题 (每题0.5分，共5分)

1. ( ) W模型优于V模型，因为W模型强调测试尽早介入，测试对象包括需求和设计文档。

::: details 答案与解析
**答案**：√

**解析**：W模型确实优于V模型，因为W模型强调测试尽早介入，在需求分析阶段就开始"测试需求"（审查文档），在设计阶段就开始"测试设计"。测试对象不仅包括程序，还包括需求文档和设计文档，能够早期发现缺陷，降低修复成本。
:::

2. ( ) 判定覆盖(Branch Coverage)一定能包含语句覆盖(Statement Coverage)。

::: details 答案与解析
**答案**：√

**解析**：判定覆盖要求每个判定的真假分支至少执行一次，这意味着所有可执行语句都会被覆盖到（因为要执行真假分支，必须经过所有语句）。因此判定覆盖一定包含语句覆盖。逻辑覆盖的强弱顺序：语句 < 判定 < 条件 < 判定/条件 < 条件组合 < 路径。
:::

3. ( ) 只要做好了单元测试，集成测试就可以省略。

::: details 答案与解析
**答案**：×

**解析**：即使单元测试做得很好，集成测试也不能省略。因为单元测试只测试单个模块，而集成测试测试模块之间的接口和交互，可能发现接口错误、数据传递错误、模块间不兼容等问题。这些问题是单元测试无法发现的。
:::

4. ( ) 测试用例的预期结果必须是确定的，不能是模糊的描述。

::: details 答案与解析
**答案**：×

**解析**：虽然大多数情况下预期结果应该是确定的，但在某些特殊场景下（如AI/大模型输出、性能测试的响应时间范围），预期结果可能是一个范围或模糊描述。但考试中，通常要求预期结果是确定的，便于比对。
:::

5. ( ) 桩模块(Stub)的主要作用是模拟被测模块的上级调用者。

::: details 答案与解析
**答案**：×

**解析**：桩模块(Stub)的主要作用是模拟被测模块的**下级**（被调用者），而不是上级。驱动模块(Driver)才是模拟上级调用者。记忆口诀：Stub是树桩，长在下面不动（模拟下级）。
:::

6. ( ) 缺陷报告中必须包含重现步骤(Steps to Reproduce)，否则开发人员可能无法修复。

::: details 答案与解析
**答案**：√

**解析**：缺陷报告必须包含重现步骤，这是缺陷报告的核心要素。没有重现步骤，开发人员无法复现问题，也就无法定位和修复缺陷。完整的缺陷报告还应包括：标题、严重性、优先级、环境信息、截图等。
:::

7. ( ) 健壮性测试(Robustness Testing)是在边界值分析的基础上，增加了对无效边界值的测试。

::: details 答案与解析
**答案**：√

**解析**：健壮性测试是在标准边界值测试（5点法：最小值、略大于最小值、正常值、略小于最大值、最大值）的基础上，增加了对无效边界值的测试（如最小值-1、最大值+1），形成7点法。目的是测试程序对异常输入的容错能力。
:::

8. ( ) 软件可靠性是指软件在规定的条件下和规定的时间内完成规定功能的能力。

::: details 答案与解析
**答案**：√

**解析**：这是软件可靠性的标准定义。可靠性是ISO 9126质量模型中的一个重要特性，关注软件在特定环境下的稳定性和可用性。
:::

9. ( ) Beta测试是由软件开发人员在开发环境下进行的测试。

::: details 答案与解析
**答案**：×

**解析**：Beta测试（β测试）是由**用户或客户**在**实际使用环境**下进行的测试，属于用户方测试（第二方测试）。Alpha测试才是由开发人员在开发环境下进行的测试。
:::

10. ( ) 判定表中的每一列对应一个测试用例。

::: details 答案与解析
**答案**：√

**解析**：判定表的每一列（规则）对应一个测试用例。这是判定表法的核心特点。判定表法适用于输入条件之间有复杂逻辑组合关系的场景。
:::

1. **以下关于H模型的描述，正确的是：**
   A. 测试准备好就可以开始，不依赖于开发阶段
   B. 测试必须在编码完成后开始
   C. 测试与开发是串行的

**解析**：H模型的核心特点是：测试是**独立的流程**，只要"测试就绪点(Test Ready Point)"到了，测试就可以开始，不依赖于开发阶段的完成。测试可以与其他流程（开发）并发进行，强调测试的灵活性和微循环。

2. **如果一个判定中的条件是 `A > 1 OR B == 0`，要满足条件覆盖，需要：**
   A. A>1(真), A>1(假), B==0(真), B==0(假) 各出现一次
   B. 整个表达式真假各一次
   C. 执行所有语句
   D. 覆盖所有路径

::: details 答案与解析
**答案**：A

**解析**：条件覆盖要求**每个子条件**的真假至少执行一次。对于 `A > 1 OR B == 0`，有两个子条件：`A>1` 和 `B==0`。因此需要 A>1(真)、A>1(假)、B==0(真)、B==0(假) 各出现一次。选项B是判定覆盖的要求，不是条件覆盖。

4. **某软件需求规格说明书中规定：员工年龄范围是 [18, 60]。使用边界值分析法，应选取的测试数据是：**
   A. 18, 60
   B. 17, 18, 60, 61
   C. 17, 18, 30, 60, 61
   D. 18, 19, 59, 60

::: details 答案与解析
**答案**：C

**解析**：边界值分析法采用**5点法**：最小值、略大于最小值、正常值、略小于最大值、最大值。对于范围[18, 60]，应选取：17（最小值-1）、18（最小值）、30（正常值）、60（最大值）、61（最大值+1）。选项B缺少正常值，选项D缺少边界外的值。
:::

5. **以下哪种逻辑覆盖标准最强？**
   A. 语句覆盖
   B. 判定覆盖
   C. 条件覆盖
   D. 路径覆盖

::: details 答案与解析
**答案**：D

**解析**：逻辑覆盖标准的强弱顺序是：**语句覆盖 < 判定覆盖 < 条件覆盖 < 判定/条件覆盖 < 条件组合覆盖 < 路径覆盖**。路径覆盖是最强的，因为它要求覆盖所有可能的执行路径，但如果有循环，路径覆盖可能无法完全实现。

12. **若代码行数为1000行，发现了5个Bug，则缺陷密度为：**

### 1. 黑盒测试：自动售货机（判定表法）

**题目描述**：
有一个处理单价为1元5角饮料的自动售货机。
- 输入条件：
  - C1: 投入1元5角硬币
  - C2: 投入2元硬币
  - C3: 按"可乐"按钮
  - C4: 按"雪碧"按钮
- 输出动作：
  - A1: 送出饮料
  - A2: 退还5角硬币
- 约束：
  - C1与C2互斥（一次只能投一种）
  - C3与C4互斥（一次只能按一种）

**要求**：
1. 补全判定表中的空缺项（特别是动作列）。（5分）
   - **提示**：重点考虑投2元时，既要出饮料，又要退钱的情况。
2. 将判定表中的**规则3（投2元，按可乐）**转化为一个标准的测试用例（包含ID、输入、预期输出）。（5分）

::: details 答案与解析
**答案**：

**1. 判定表**：

| 条件桩 / 动作桩 | 规则1 | 规则2 | 规则3 | 规则4 | 规则5 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **条件** | | | | | |
| C1 (投1.5元) | 1 | 1 | 0 | 0 | 0 |
| C2 (投2元) | 0 | 0 | 1 | 1 | 0 |
| C3 (按可乐) | 1 | 0 | 1 | 0 | 0 |
| C4 (按雪碧) | 0 | 1 | 0 | 1 | 0 |
| **动作** | | | | | |
| A1 (出饮料) | **1** | **1** | **1** | **1** | 0 |
| A2 (退5角) | 0 | 0 | **1** | **1** | 0 |

**逻辑推导**：
- **规则1**：投1.5元 + 按可乐 → 正好1.5元，出饮料，不退钱
- **规则2**：投1.5元 + 按雪碧 → 正好1.5元，出饮料，不退钱
- **规则3**：投2元 + 按可乐 → 多投0.5元，**既要出饮料，又要退5角**（关键点！）
- **规则4**：投2元 + 按雪碧 → 多投0.5元，**既要出饮料，又要退5角**
- **规则5**：没投币或没按按钮 → 什么都不做

**关键点**：规则3和规则4的A2必须填1，这是最容易漏填的地方！

**2. 测试用例**：

| 用例ID | 输入 | 预期输出 |
| :--- | :--- | :--- |
| TC_03 | 投入2元硬币，按下"可乐"按钮 | 送出可乐饮料，退还5角硬币 |

**注意**：预期输出必须包含两项：出饮料和退钱，不能只写一项。

### 2. 白盒测试：逻辑覆盖与路径分析

**代码片段**：
```java
public void check(int a, int b, int c) {
    if (a > 0 && b > 0) { // P1
        c = c / a;
    }
    if (a > 1 || c > 1) { // P2
        c = c + 1;
    }
}
```

**要求**：
1. 计算该代码段的**圈复杂度 V(G)**，并写出计算过程（三个公式验算）。（4分）
   - *提示*：复合条件必须拆分！
2. 设计一组测试用例，满足**判定覆盖 (Decision Coverage)**。（3分）
   - 每个判定（P1, P2）的真假分支至少执行一次。
3. 设计一组测试用例，满足**条件覆盖 (Condition Coverage)**。（3分）
   - 每个子条件（a>0, b>0, a>1, c>1）的真假至少执行一次。

::: details 答案与解析
**答案**：

**1. 圈复杂度计算**：

**控制流图**（复合条件必须拆分）：
- 判定节点：①(a>0), ②(b>0), ③(a>1), ④(c>1)，共4个
- 使用公式3：V(G) = P + 1 = 4 + 1 = **5**

**验算**：
- 公式1：V(G) = 区域数 = 5
- 公式2：V(G) = E - N + 2（需要数边和节点）
- 公式3：V(G) = P + 1 = 4 + 1 = 5

**答案：V(G) = 5**

**2. 判定覆盖测试用例**：

需要保证每个判定（P1, P2）的真假分支至少执行一次：

| 用例ID | 输入 (a, b, c) | 覆盖的判定分支 |
| :--- | :--- | :--- |
| TC_01 | a=1, b=1, c=1 | P1(假), P2(假) |
| TC_02 | a=2, b=1, c=0 | P1(真), P2(真) |

**说明**：
- TC_01：a=1不满足a>0（假），进入P1的假分支；a=1不满足a>1且c=0不满足c>1，进入P2的假分支
- TC_02：a=2>0且b=1>0，进入P1的真分支；a=2>1，进入P2的真分支（短路）

**3. 条件覆盖测试用例**：

需要保证每个子条件（a>0, b>0, a>1, c>1）的真假至少执行一次：

| 用例ID | 输入 (a, b, c) | 覆盖的条件 |
| :--- | :--- | :--- |
| TC_03 | a=0, b=0, c=0 | a>0(假), b>0(假), a>1(假), c>1(假) |
| TC_04 | a=2, b=2, c=2 | a>0(真), b>0(真), a>1(真), c>1(真) |

**说明**：条件覆盖要求每个子条件的真假都出现，不要求组合。

## 四、案例分析题 (共13分)

### 1. Ariane 5 火箭事故分析 (7分)
**背景**：1996年，Ariane 5火箭在发射后40秒爆炸。调查显示，其惯性基准系统直接重用了Ariane 4的软件。Ariane 5的水平速度比Ariane 4快得多，导致一个64位浮点数在转换为16位有符号整数时发生溢出，引发系统异常。

**问题**：
1. 导致事故的技术原因是什么？（2分）
2. 从管理和测试的角度看，根本原因是什么？（软件重用方面）（3分）
3. 这个案例对我们在"旧代码新用"时有什么启示？（2分）

::: details 答案与解析
**答案**：

**1. 技术原因**：

**64位浮点数转换为16位有符号整数时发生溢出**。

Ariane 5的水平速度比Ariane 4快得多，导致惯性基准系统计算出的速度值超出了16位有符号整数的表示范围（-32768到32767），发生溢出，引发系统异常，最终导致火箭爆炸。

**2. 根本原因（管理/测试）**：

- **直接重用了Ariane 4的代码**，未考虑新环境的变化
- **Ariane 5的水平速度超出了原代码的适用范围**，但未在新的环境下进行测试
- **在新的环境下完全没有进行测试**，假设旧代码在新环境下仍然适用

这是典型的"代码重用未测试"问题。

**3. 启示**：

- **代码重用必须重新测试**：特别是当环境、前提、假设改变时，必须在新环境下进行充分测试
- **回归测试的重要性**：重用代码时，必须进行回归测试，确保代码在新环境下仍然正确工作
- **不能假设旧环境下的测试结果适用于新环境**：环境变化可能导致代码失效
- **测试计划中应包含代码重用的测试策略**：对于重用的代码，应制定专门的测试计划

**采分点**：
- 必须提到"浮点转整数溢出"和"代码重用未测试"
- 必须结合测试理论（回归测试、环境变化）
- 使用专业术语

### 2. 测试管理：人员配比问题 (6分)
**背景**：某创业公司开发一款复杂的金融交易系统，团队配置如下：
- 开发人员：20人
- 测试人员：1人（兼职运维）
- 项目周期：3个月
结果上线后，系统频繁崩溃，资金数据对不上，客户投诉不断。CEO认为是测试人员能力不行，打算开除测试人员。

2. 作为一个专业的测试经理，你建议如何调整团队结构？

::: details 答案与解析
**答案**：

**1. 存在的问题**：

根据E.Dustin的测试团队配比建议：
- **开发:测试 = 20:1**，配比严重失调
- **金融交易系统属于高风险系统**，需要更高的测试投入
- **测试人员只有1人且兼职运维**，无法保证测试质量
- **项目周期只有3个月**，时间紧迫，测试人员不足会导致测试不充分

- 操作系统类：开发:测试 ≈ 2:1
- 应用平台类：开发:测试 ≈ 1:1
- 特定应用类（高风险）：开发:测试 ≈ 1:2~1:4

金融交易系统属于高风险系统，应该接近1:1或至少3:1，而不是20:1。

**2. 调整建议**：

- **增加测试人员**：至少增加到5-7人（开发:测试 ≈ 3:1），理想情况下增加到10人（1:1）
- **专职测试人员**：测试人员不能兼职运维，必须专职测试
- **测试团队结构**：
  - 测试经理1人（统筹、计划）
  - 测试设计人员2-3人（资深，负责用例设计）
  - 测试执行人员3-4人（执行测试、发现问题）
  - 自动化测试工程师1人（工具和脚本）
- **延长测试时间**：如果时间不够，建议申请延期发布，不能牺牲质量
- **加强测试培训**：对测试人员进行金融业务培训，提高测试专业性

**采分点**：
- 必须提到配比失调（20:1）
- 必须提到高风险系统需要更高测试投入
- 必须给出具体的调整建议（人数、结构）
