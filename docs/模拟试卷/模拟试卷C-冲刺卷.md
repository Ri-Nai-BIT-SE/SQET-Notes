## 一、判断题 (每题0.5分，共5分)

1. ( ) 如果测试用例设计得足够好，是可以发现所有软件缺陷的。
2. ( ) 国际化(I18N)是本地化(L10N)的基础和前提。
3. ( ) 自动化测试的主要目的是发现新的Bug，而不是回归测试。
4. ( ) 在进行单元测试时，必须将被测模块与其他模块隔离开，因此需要驱动和桩。
5. ( ) 语句覆盖(Statement Coverage)是最强的逻辑覆盖标准。
6. ( ) 软件测试文档（如测试计划、测试用例）也是测试的对象。
7. ( ) 缺陷清除率 (Defect Removal Efficiency) 越高，说明测试效果越好。
8. ( ) 负载测试(Load Testing)和压力测试(Stress Testing)是一回事。
9. ( ) 测试环境的配置错误（如数据库版本不一致）不会影响测试结果的有效性。
10. ( ) 软件质量保证(QA)主要关注的是过程，而软件质量控制(QC)主要关注的是产品。

## 三、设计大题 (每题10分，共20分)

### 1. 黑盒测试：NextDate问题 (边界值分析)

**题目描述**：
程序输入三个整数：年(Y)、月(M)、日(D)。
- 年：[1900, 2050]
- 月：[1, 12]
- 日：[1, 31]
程序输出下一天的日期。

**要求**：
1. 请分析该问题中的**边界值**（最小值、略大于最小值、正常值、略小于最大值、最大值）。
2. 针对**"日(D)"**这个变量，结合月(M)的取值，设计一组能够覆盖所有月份天数边界（28/29/30/31日）的测试用例。
   - 需包含闰年2月29日的情况。
   - **格式要求**：ID | 输入(Y,M,D) | 预期输出 | 说明

::: details 答案与解析
**答案**：

**1. 边界值分析**：

- **年(Y)**：[1900, 2050]
  - 最小值：1900
  - 略大于最小值：1901
  - 正常值：2000
  - 略小于最大值：2049
  - 最大值：2050

- **月(M)**：[1, 12]
  - 最小值：1
  - 略大于最小值：2
  - 正常值：6
  - 略小于最大值：11
  - 最大值：12

- **日(D)**：[1, 31]（但需结合月份）
  - 最小值：1
  - 略大于最小值：2
  - 正常值：15
  - 略小于最大值：30（小月）或30（大月）
  - 最大值：28/29（2月）或30（小月）或31（大月）

**2. 测试用例设计**（覆盖月份天数边界）：

| 用例ID | 输入(Y,M,D) | 预期输出 | 说明 |
| :--- | :--- | :--- | :--- |
| TC_01 | 2000, 1, 31 | 2000, 2, 1 | 大月31日转下月1日 |
| TC_02 | 2000, 2, 28 | 2000, 2, 29 | 平年2月28日转29日 |
| TC_03 | 2000, 2, 29 | 2000, 3, 1 | 闰年2月29日转3月1日 |
| TC_04 | 2001, 2, 28 | 2001, 3, 1 | 平年2月28日转3月1日 |
| TC_05 | 2000, 4, 30 | 2000, 5, 1 | 小月30日转下月1日 |
| TC_06 | 2000, 12, 31 | 2001, 1, 1 | 跨年12月31日转1月1日 |

**关键点**：
- ✅ 必须覆盖2月28/29日（平年/闰年）
- ✅ 必须覆盖大月31日转下月1日
- ✅ 必须覆盖小月30日转下月1日
- ✅ 必须覆盖跨年情况（12.31 → 1.1）
- ✅ 闰年判断：能被4整除但不能被100整除，或能被400整除
:::

### 2. 白盒测试：综合逻辑覆盖

**代码片段**：
```c
if ( (A > 1) && (B == 0) ) { // 判定1
    X = X / A;
}
if ( (A == 2) || (X > 1) ) { // 判定2
    X = X + 1;
}
```

**要求**：
1. 画出控制流图。（3分）
2. 设计最少数量的测试用例，满足**条件覆盖 (Condition Coverage)**。（每个子条件的真假都出现一次）（4分）
   - 子条件：`A>1`, `B==0`, `A==2`, `X>1`
3. 计算圈复杂度。（3分）

::: details 答案与解析
**答案**：

**1. 控制流图**：

```
节点①: A > 1?
  ├─ T → 节点②: B == 0?
  │         ├─ T → 节点③: X = X / A → 节点④
  │         └─ F → 节点④
  └─ F → 节点④: A == 2?
              ├─ T → 节点⑤: X = X + 1 → 节点⑥
              └─ F → 节点⑦: X > 1?
                        ├─ T → 节点⑤ → 节点⑥
                        └─ F → 节点⑥
节点⑥: 结束
```

**注意**：复合条件必须拆分！

**2. 条件覆盖测试用例**：

需要保证每个子条件（A>1, B==0, A==2, X>1）的真假至少执行一次：

| 用例ID | 输入 (A, B, X) | 覆盖的条件 | 说明 |
| :--- | :--- | :--- | :--- |
| TC_01 | A=0, B=1, X=0 | A>1(假), B==0(假), A==2(假), X>1(假) | 所有条件为假 |
| TC_02 | A=2, B=0, X=2 | A>1(真), B==0(真), A==2(真), X>1(真) | 所有条件为真 |

**最少用例数**：2个（可以覆盖所有子条件的真假）

**3. 圈复杂度计算**：

- 判定节点：①(A>1), ②(B==0), ④(A==2), ⑦(X>1)，共4个
- V(G) = P + 1 = 4 + 1 = **5**

**验算**：
- 公式1：V(G) = 区域数 = 5
- 公式3：V(G) = P + 1 = 4 + 1 = 5
:::

## 四、案例分析题 (共13分)

### 1. 12306 早期崩溃分析 (6分)
**背景**：12306 刚上线时，每逢春运抢票高峰期必崩溃，出现页面打不开、扣钱不出票、余票显示错误等问题。

**问题**：
1. 从性能测试的角度，这属于什么类型的问题？（负载/压力/并发？）（2分）
2. "扣钱不出票"属于什么性质的缺陷？（数据一致性/事务）（2分）
3. 如果你是测试经理，你会重点安排哪些类型的测试？（2分）

::: details 答案与解析
**答案**：

**1. 性能问题类型**：

这属于**高并发问题**（Concurrency Testing）。

在春运抢票高峰期，大量用户同时访问系统（高并发），导致系统无法承受，出现崩溃。这不仅仅是负载测试（正常负载）或压力测试（极限负载），而是**并发测试**，关注系统在大量用户同时操作时的表现。

**2. 缺陷性质**：

"扣钱不出票"属于**事务一致性**问题（Transaction Consistency）。

这是典型的**事务完整性**缺陷：支付操作和出票操作应该作为一个原子事务，要么全部成功，要么全部失败。但系统在扣钱后未能成功出票，导致数据不一致。这违反了ACID原则中的原子性(Atomicity)和一致性(Consistency)。

**3. 重点测试类型**：

- **并发测试**：模拟大量用户同时抢票的场景
- **压力测试**：测试系统在极限负载下的表现，找出崩溃点
- **负载测试**：测试系统在正常和稍高负载下的表现
- **事务完整性测试**：确保支付和出票操作的原子性
- **数据一致性测试**：确保扣钱和出票数据的一致性
- **性能测试**：测试响应时间、吞吐量等指标

**采分点**：
- 必须识别出"高并发"问题
- 必须提到"事务一致性"或"原子性"
- 必须列出具体的测试类型（并发、压力、事务测试）
- 使用专业术语
:::

### 2. 缺陷趋势分析 (7分)
**背景**：在项目发布前一周，你发现缺陷趋势图（Bug Trend）显示：**新建Bug数量（New）持续居高不下，且修复Bug数量（Fixed）远低于新建数量**。

**问题**：
1. 这条 "NOB (Number of Open Bugs)" 曲线说明系统处于什么状态？（稳定/不稳定）（2分）
2. 作为一个负责任的测试经理，你会建议按时发布吗？为什么？（2分）
3. 你会采取什么紧急措施？（3分）

::: details 答案与解析
**答案**：

**1. 系统状态**：

系统处于**极不稳定**状态。

**NOB曲线分析**：
- 新建Bug数量持续居高不下，说明系统还存在大量问题
- 修复Bug数量远低于新建数量，说明修复速度跟不上发现速度
- 未关闭的Bug数量持续上升，说明系统质量在恶化

这表明系统**质量失控**，不适合发布。

**2. 发布建议**：

**坚决反对按时发布**。

**原因**：
- 系统质量不稳定，存在大量未修复的缺陷
- 发布后可能导致严重的生产事故
- 修复成本会急剧上升（发布后修复成本可能是发布前的100倍）
- 会影响用户体验和公司声誉

**建议**：申请延期发布，优先修复严重缺陷，待系统稳定后再发布。

**3. 紧急措施**：

- **停止新功能开发**：集中所有开发资源修复现有缺陷
- **优先修复严重缺陷**：按照严重性（致命、严重、一般）排序，优先修复致命和严重缺陷
- **增加测试人员**：如果测试人员不足，临时增加测试人员，加快验证速度
- **延长工作时间**：如果时间紧迫，可以安排加班，但不能牺牲质量
- **召开紧急会议**：与项目经理、开发经理、产品经理开会，评估风险，制定修复计划
- **重新评估发布标准**：明确发布的质量标准（如：致命缺陷0个，严重缺陷不超过X个）
- **加强回归测试**：修复缺陷后，必须进行充分的回归测试，防止引入新缺陷

**采分点**：
- 必须识别出"不稳定"状态
- 必须明确"反对发布"
- 必须给出具体的紧急措施（至少3条）
- 使用专业术语（NOB曲线、质量失控、回归测试）
:::
