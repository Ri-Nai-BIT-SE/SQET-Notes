## 一、判断题 (每题0.5分，共5分)

1. ( ) 软件测试的目的是为了证明程序没有错误。

::: details 答案与解析
**答案**：×

**解析**：根据Myers的观点，软件测试的目的是**发现错误**，而不是证明程序没有错误。这是测试的"证伪"观点，也是主流观点。测试无法通过有限测试证明程序完全正确。

📖 **相关知识点**：
- [软件测试的定义与目的](/第一部分-基础概念与原理/1-软件测试的定义与目的#myers迈尔斯的观点1979)
- [Myers的观点详解](/第一部分-基础概念与原理/1-软件测试的定义与目的#测试的辩证观点)
:::

2. ( ) 软件测试工作应该在代码编写完成后才开始。

::: details 答案与解析
**答案**：×

**解析**：测试应该**尽早开始**，从需求分析阶段就应该介入（如需求评审）。W模型和H模型都强调测试尽早介入，而不是等到编码完成后才开始。

📖 **相关知识点**：
- [软件测试模型 - W模型](/第二部分-测试分类与过程模型/2-软件测试模型#-w-模型-w-model--双v模型)
- [软件测试模型 - H模型](/第二部分-测试分类与过程模型/2-软件测试模型#-h-模型-h-model--主推模型)
- [测试尽早介入的重要性](/第一部分-基础概念与原理/4-缺陷成本)
:::

3. ( ) 只有测试人员才对软件质量负责，开发人员只负责写代码。

::: details 答案与解析
**答案**：×

**解析**：软件质量是**全员责任**，不仅仅是测试人员的责任。开发人员、项目经理、测试人员都应该对质量负责。测试人员负责发现缺陷，开发人员负责修复缺陷，PM负责协调资源。
:::

4. ( ) 在V模型中，验收测试的依据是用户需求说明书。

::: details 答案与解析
**答案**：√

**解析**：在V模型中，验收测试对应的是用户需求阶段，其依据是用户需求说明书或任务书。V模型的对应关系是：验收测试 ↔ 用户需求，系统测试 ↔ 需求分析，集成测试 ↔ 概要设计，单元测试 ↔ 详细设计/编码。
:::

5. ( ) 软件缺陷大部分来自于编码阶段的错误。

::: details 答案与解析
**答案**：×

**解析**：**需求规格说明书**是缺陷产生的最大来源（约50%以上），而不是编码阶段。缺陷可能来自需求、设计、编码、测试等各个阶段，但需求阶段的缺陷修复成本最高。
:::

6. ( ) 验证(Verification)关注的是"Are we building the product right?"（过程正确性）。

::: details 答案与解析
**答案**：√

**解析**：验证(Verification)关注的是过程正确性，即"我们是在正确地开发产品吗？"，检查是否符合上一阶段的文档要求。而确认(Validation)关注的是"Are we building the right product?"（结果正确性），检查是否符合用户真实需求。
:::

7. ( ) 单元测试通常由专门的黑盒测试人员完成。

::: details 答案与解析
**答案**：×

**解析**：单元测试通常由**开发人员（程序员）**自己完成，而不是专门的测试人员。因为单元测试需要了解代码内部结构，属于白盒测试范畴。
:::

8. ( ) 软件本地化仅仅是将界面语言翻译成当地语言。

::: details 答案与解析
**答案**：×

**解析**：软件本地化不仅包括语言翻译，还包括：日期格式、货币单位、度量衡、文化禁忌、界面布局（如阿拉伯语从右向左）等。这是一个综合性的适配过程。
:::

9. ( ) 自底向上集成测试需要编写驱动模块(Driver)。

::: details 答案与解析
**答案**：√

**解析**：自底向上集成测试需要编写**驱动模块(Driver)**来模拟被测模块的上级调用者。记忆口诀：Driver是司机，坐在上面开车（模拟上级）。而自顶向下集成测试需要编写桩模块(Stub)来模拟下级。
:::

10. ( ) 开发人员修复Bug后，可以自己将Bug状态直接改为Closed。

::: details 答案与解析
**答案**：×

**解析**：**只有测试人员有权关闭(Close)缺陷**。开发人员修复后只能将状态改为"Fixed/Resolved"，然后由测试人员验证修复是否成功，验证通过后才能关闭。这是缺陷管理流程的核心规则。
:::

## 二、单项选择题 (每题1分，共22分)

1. **根据Myers的观点，软件测试的核心目的是：**
   A. 证明程序无错
   B. 发现程序中的错误
   C. 评估程序员的能力
   D. 保证项目进度

::: details 答案与解析
**答案**：B

**解析**：Myers在1979年出版的《软件测试的艺术》中提出，测试的目的是**发现错误**，而不是证明程序无错。这是测试的"证伪"观点，也是主流观点。考试中大多数选择题都基于Myers的观点。
:::

2. **以下哪个模型强调测试与开发是并行进行的？**
   A. 瀑布模型
   B. V模型
   C. W模型
   D. 螺旋模型

::: details 答案与解析
**答案**：C

**解析**：**W模型（双V模型）**强调测试与开发并行进行。在需求分析阶段，测试人员就开始"测试需求"（审查文档）；在设计阶段，就开始"测试设计"。V模型是串行的，测试介入较晚。

3. **第三方测试的核心特点是：**
   A. 成本低
   B. 效率高
   C. 独立、客观、公正
   D. 熟悉代码

::: details 答案与解析
**答案**：C

**解析**：第三方测试的核心特点是**独立、客观、公正**。第三方测试必须在技术、管理、财务上完全独立于开发方和用户方，这样才能保证测试结果的权威性和公正性。

5. **以下关于缺陷生命周期的描述，正确的是：**
   A. 测试人员发现 -> 开发人员修复 -> 开发人员关闭
   B. 测试人员发现 -> 开发人员修复 -> 测试人员验证 -> 测试人员关闭
   C. 开发人员发现 -> 开发人员修复 -> 开发人员关闭
   D. 项目经理发现 -> 开发人员修复 -> 项目经理关闭

::: details 答案与解析
**答案**：B

**解析**：标准的缺陷生命周期流程是：测试人员发现(Open) -> 分配给开发人员(Assigned) -> 开发人员修复(Fixed) -> **测试人员验证(Verified)** -> **测试人员关闭(Closed)**。只有测试人员有权关闭缺陷，这是必考点。

6. **随着软件生命周期的推移，修复缺陷的成本呈什么趋势？**
   A. 线性增长
   B. 保持不变
   C. 指数级增长
   D. 逐渐降低

::: details 答案与解析
**答案**：C

**解析**：修复缺陷的成本随着发现时间的推迟呈**指数级（几何级数）增长**。需求阶段发现成本最低，发布后发现成本可能达到需求阶段的100倍以上。因此测试应尽早开始。

7. **以下属于动态测试的是：**
   A. 代码走查
   B. 需求评审
   C. 运行程序
   D. 文档审查

::: details 答案与解析
**答案**：C

**解析**：**动态测试**是指运行程序进行测试，包括功能测试、性能测试等。而代码走查、需求评审、文档审查都属于**静态测试**（不运行程序，只审查文档和代码）。
:::

8. **如果系统的CPU利用率长期维持在95%以上，最可能存在的问题是：**
   A. 内存泄漏
   B. CPU瓶颈
   C. 磁盘I/O瓶颈
   D. 网络拥塞

::: details 答案与解析
**答案**：B

**解析**：CPU利用率长期维持在95%以上，说明CPU资源被大量占用，最可能是**CPU瓶颈**。可能的原因包括：算法效率低、计算量大、需要优化算法或升级CPU。内存泄漏的特征是内存占用持续上升且不下降。
:::

9. **验收测试的主要参与者是：**
   A. 开发人员
   B. 测试人员
   C. 用户或客户
   D. 系统管理员

::: details 答案与解析
**答案**：C

**解析**：验收测试（Acceptance Testing）的主要参与者是**用户或客户**，这是用户方测试（第二方测试/β测试）。目的是确认软件是否符合用户的实际需求，是否可以在实际环境中使用。
:::

10. **以下不属于软件质量模型（ISO 9126）特性的是：**
    A. 功能性
    B. 易用性
    C. 成本
    D. 可靠性

::: details 答案与解析
**答案**：C

**解析**：ISO 9126软件质量模型包含六个特性：功能性、可靠性、易用性、效率、维护性、可移植性。**成本**不属于质量特性，而是项目管理的范畴。

13. **关于等价类划分，以下说法错误的是：**
    A. 有效等价类是指对于程序的规格说明来说是合理的、有意义的输入数据构成的集合
    B. 设计测试用例时，应尽可能多地覆盖尚未覆盖的有效等价类
    C. 设计测试用例时，应尽可能多地覆盖尚未覆盖的无效等价类
    D. 无效等价类用于检查程序是否对异常输入进行了处理

::: details 答案与解析
**答案**：C

**解析**：设计测试用例时，**无效等价类应该每次只覆盖一个**，不能一次覆盖多个无效类，否则可能导致错误屏蔽（程序在第一个错误处就退出，后面的错误没测到）。有效等价类可以合并测试，提高效率。

    A. 黑盒测试
    B. 白盒测试

    D. 静态测试

15. **以下哪个不属于兼容性测试的范畴？**
    A. 不同浏览器的兼容
    B. 不同操作系统的兼容

    D. 代码逻辑的正确性

**答案**：D

**解析**：兼容性测试包括：硬件兼容、操作系统兼容、浏览器兼容、数据库版本兼容、分辨率兼容等。**代码逻辑的正确性**属于功能测试的范畴，不属于兼容性测试。
:::

    A. 用户需求

    D. 详细设计

**答案**：C

**解析**：V模型的对应关系是：验收测试 ↔ 用户需求，系统测试 ↔ 需求分析，**集成测试 ↔ 概要设计**，单元测试 ↔ 详细设计/编码。这是必背的对应关系表。

19. **缺陷密度是指：**
    A. 缺陷数量 / 代码行数
    B. 修复缺陷的时间
    C. 发现缺陷的人数
    D. 严重缺陷的比例

::: details 答案与解析
**答案**：A

**解析**：**缺陷密度 = 缺陷数量 / 代码行数**（或功能点）。这是衡量代码质量的重要指标，单位通常是"个/KLOC"（每千行代码的缺陷数）。
:::

20. **关于自动化测试，以下说法正确的是：**
    A. 自动化测试可以完全取代手工测试
    B. 自动化测试总能发现新Bug
    C. 回归测试适合使用自动化测试
    D. 任何项目都应该从一开始就实施自动化测试

::: details 答案与解析
**答案**：C

**解析**：**回归测试适合使用自动化测试**，因为回归测试需要重复执行相同的用例，自动化可以提高效率。但自动化测试不能完全取代手工测试（如探索性测试），也不是所有项目都适合自动化（需要评估ROI）。
:::

21. **软件本地化 (Localization) 不包括：**
    A. 翻译界面文本
    B. 调整日期和时间格式
    C. 优化算法效率
    D. 调整货币单位

::: details 答案与解析
**答案**：C

**解析**：软件本地化包括：翻译界面文本、调整日期格式、调整货币单位、调整度量衡、文化禁忌适配、界面布局调整等。**优化算法效率**属于性能优化，不属于本地化范畴。
:::

22. **以下哪个测试阶段最适合进行性能测试？**
    A. 单元测试
    B. 集成测试
    C. 系统测试
    D. 验收测试

::: details 答案与解析
**答案**：C

**解析**：**系统测试阶段**最适合进行性能测试，因为此时系统已经完整集成，可以测试整个系统的性能表现（响应时间、吞吐量、资源利用率等）。单元测试和集成测试阶段系统还不完整，不适合性能测试。
:::

## 三、设计大题 (每题10分，共20分)

### 1. 黑盒测试：三角形问题（等价类划分法）

**题目描述**：
输入三个整数 a, b, c 分别作为三角形的三条边，取值范围是 [1, 100]。程序判断是等边、等腰、普通三角形，还是构不成三角形。

**要求**：
1. 填空补全等价类划分表（4分）。

### 2. 白盒测试：控制流图与圈复杂度

**代码片段**：
```c
void test(int A, int B) {
    if (A > 5 && B == 0) {  // 复合条件
        X = X / A;
    }
    if (A == 2 || X > 1) {  // 复合条件
        X = X + 1;
    }
    printf("%d", X);
}
```

**要求**：
1. 画出该代码的控制流图（注意：复合条件必须拆分）。（4分）
2. 计算圈复杂度 V(G)。（3分）
3. 列出所有的独立路径。（3分）

::: details 答案与解析
**答案**：

**1. 控制流图**：

```
节点①: A > 5?
  ├─ T → 节点②: B == 0?
  │         ├─ T → 节点③: X = X / A → 节点④
  │         └─ F → 节点④
  └─ F → 节点④: A == 2?
              ├─ T → 节点⑤: X = X + 1 → 节点⑥
              └─ F → 节点⑦: X > 1?
                        ├─ T → 节点⑤ → 节点⑥
                        └─ F → 节点⑥
节点⑥: printf (结束)
```

**注意**：
- 复合条件 `A > 5 && B == 0` 必须拆分成两个判定节点（①和②）
- 复合条件 `A == 2 || X > 1` 必须拆分成两个判定节点（④和⑦）
- 节点④是第一个IF结束后的汇聚节点

**2. 圈复杂度计算**：

使用三个公式验算：

- **公式1**：V(G) = 区域数 = 5（包括外部区域）
- **公式2**：V(G) = E - N + 2（需要数边和节点）
- **公式3**：V(G) = P + 1 = 4 + 1 = 5

判定节点有：①(A>5), ②(B==0), ④(A==2), ⑦(X>1)，共4个。

**答案：V(G) = 5**

**3. 独立路径**（需要5条）：

- **路径1**：①(F) → ④(A!=2) → ⑦(X<=1) → ⑥
  - 测试用例：A=3, B=1, X=0

- **路径2**：①(T) → ②(T) → ③ → ④(A!=2) → ⑦(X<=1) → ⑥
  - 测试用例：A=10, B=0, X=1（执行除法后X=0）

- **路径3**：①(T) → ②(F) → ④(A!=2) → ⑦(X>1) → ⑤ → ⑥
  - 测试用例：A=10, B=1, X=2

- **路径4**：①(F) → ④(A==2) → ⑤ → ⑥
  - 测试用例：A=2, B=1, X=0

- **路径5**：①(F) → ④(A!=2) → ⑦(X>1) → ⑤ → ⑥
  - 测试用例：A=3, B=1, X=2

**关键点**：
- ✅ 复合条件必须拆分，否则圈复杂度算错，后面全扣分
- ✅ 每条路径必须包含至少一条之前路径没有走过的边
- ✅ 用三个公式验算圈复杂度，确保结果一致

### 1. Therac-25 事故分析 (6分)
**背景**：Therac-25 是一款放射治疗仪，在1985-1987年间发生了严重的软件故障，导致多名患者死亡。调查发现，该系统直接重用了旧型号 T-20 的部分代码，但去掉了原本 T-20 依赖的硬件联锁保护机制，完全依赖软件进行安全检查。

**问题**：
1. 请分析导致 Therac-25 事故的主要原因（至少两点）。
2. 从软件测试的角度，我们应该吸取什么教训？

::: details 答案与解析
**答案**：

**1. 主要原因**（至少两点）：

- **软件未包含在安全性分析中**：系统只信任硬件联锁保护，软件去掉了原本T-20依赖的硬件保护机制，完全依赖软件进行安全检查，但软件本身没有被纳入安全性分析。

- **代码重用未考虑环境变化**：直接重用了T-20的软件代码，但硬件环境变了（假设和前提改变），未在新的环境下重新测试。这是典型的"旧代码新用"问题。

**2. 教训**：

- **代码重用必须重新测试**：特别是当环境、前提、假设改变时，必须在新环境下进行充分测试，不能假设旧环境下的测试结果适用于新环境。

- **软件安全性分析的重要性**：对于安全关键系统，软件必须包含在安全性分析中，不能仅依赖硬件保护。

- **回归测试的重要性**：在重用代码时，必须进行回归测试，确保代码在新环境下仍然正确工作。

**采分点**：
- 必须提到"代码重用"和"安全性分析"
- 必须结合测试理论（回归测试、环境变化）
- 使用专业术语，不要用大白话

### 2. 性能测试分析 (7分)
**背景**：某电商网站在双十一促销期间，用户反映页面加载极慢。测试团队对系统进行了性能监控，发现如下数据：
- 并发用户数：5000
- CPU利用率：15%
- 内存利用率：40%
- 数据库连接池：已满
- 响应时间：平均 8秒

**问题**：
1. 根据上述数据，系统的瓶颈最可能在哪里？为什么？（排除 CPU 和 内存）
2. "响应时间 8秒" 符合一般的性能标准吗？请解释 "2-5-8 原则"。

::: details 答案与解析
**答案**：

**1. 系统瓶颈**：

系统的瓶颈最可能在**数据库连接池**。

**原因分析**：
- CPU利用率只有15%，说明CPU不是瓶颈
- 内存利用率只有40%，说明内存不是瓶颈
- **数据库连接池已满**，这是关键指标
- 响应时间平均8秒，说明系统处理很慢

当数据库连接池已满时，新的请求无法获取数据库连接，只能等待，导致响应时间变长。即使应用服务器的CPU和内存使用率不高，但数据库连接资源耗尽，整个系统性能就会下降。

**可能的解决方案**：
- 增加数据库连接池大小
- 优化SQL查询，减少连接占用时间
- 检查是否有连接泄漏（连接未正确释放）

**2. 响应时间标准**：

"响应时间 8秒" **不符合**一般的性能标准。

**2-5-8 原则**：
- **2秒**：优秀，用户感觉流畅
- **5秒**：良好，用户可接受
- **8秒**：无法忍受，用户会感到不耐烦，可能放弃操作

因此，8秒的响应时间已经超出了用户可接受的范围，需要优化。

**采分点**：
- 必须识别出数据库连接池是瓶颈
- 必须解释为什么（连接资源耗尽）
- 必须说明2-5-8原则的具体数值
- 使用专业术语（响应时间、瓶颈、连接池）
