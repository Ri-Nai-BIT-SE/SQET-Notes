### **Part 2: The Testing Team**

This section delves into the structure, roles, and responsibilities of a software testing team.

*   **Team Mission and Responsibility:**
    *   The primary goal is to **find all possible problems** in the software, a concept aligned with the逆向 (reverse) view of testing—proving the software is broken to gain confidence.
    *   **督促 (Urge) developers** to fix defects promptly. This involves a "whoever finds it, confirms it" process where the tester who reports a bug is also responsible for verifying its fix.
    *   **Assist in creating reasonable development plans** by providing insights into product quality.
    *   **Analyze, classify, and track defects** to provide a clear view of the product's quality status to management.
    *   **Improve the development process** by promoting better coding standards (readability, maintainability).

*   **Team Structure and Models:**
    *   **以开发为核心 (Developer-centric):** A development manager oversees developers, testers, and documentation staff.
    *   **以项目经理为核心 (Project Manager-centric):** A project manager leads separate development and testing sub-teams.
    *   **三国鼎立 (Three-way balance):** Project, development, and test managers operate as parallel leads.

*   **Team Size and Composition:**
    *   Team size depends on the project's scope, complexity, and budget. It requires estimating the testing workload, often in "man-months," similar to development estimation.
    *   A typical team includes a **Test Manager/QA Manager**, **Lab/Configuration Manager** (often overlooked, responsible for managing test environments and versions), **Internal Auditor** (for process review), **Test Lead**, **Test Designer**, and **Test Executor** (Junior Testers).

### **Part 3: Roles and Career Path in a Testing Team**

The lecture provides a detailed breakdown of the responsibilities at different career levels within a testing team.

*   **初级测试工程师 (Junior Test Engineer):**
    *   Executes pre-written test cases.
    *   Identifies and clearly describes defects.
    *   Uses simple test tools and learns basic software engineering methods.

*   **测试工程师 (Test Engineer):**
    *   Designs functional test cases based on requirements.
    *   Uses advanced test tools, including developing test scripts.
    *   Sets up simple test environments and tracks defects until closure.

*   **资深测试工程师 (Senior Test Engineer):**
    *   Responsible for testing one or more complex modules.
    *   Develops test plans and strategies for specific modules.
    *   Designs automation frameworks, develops complex test scripts, and sets up intricate test environments.
    *   Conducts performance, failover, and security testing.
    *   Reviews code, design documents, and specifications.

*   **A Day in the Life of a Microsoft Test Engineer:**
    An example is given to illustrate a typical workflow: automated builds run overnight, testers review failure reports in the morning, high-priority bugs are fixed the same day, and testers spend their day verifying fixes, running regression tests, and developing new automated test scripts.

### **Part 4: Test Requirement Analysis and Test Planning**

This final section focuses on the foundational steps of any testing project.

*   **Defining Test Objectives:**
    *   The first step is to clarify the goal of testing. Is it to assess the quality of a student project? To validate a new feature? Or to ensure a critical system is flawless?
    *   Key metrics (the "four rates") are introduced to measure success:
        1.  **测试执行率 (Test Execution Rate):** Executed cases / Planned cases.
        2.  **执行通过率 (Execution Pass Rate):** Passed cases / Executed cases.
        3.  **严重缺陷修复率 (Critical Defect Fix Rate):** Fixed critical defects / Found critical defects.
        4.  **缺陷解决率 (Defect Resolution Rate):** Resolved defects / Total defects.

*   **Test Requirement Analysis Process:**
    *   **Understand the project background and business value.**
    *   **Analyze user needs** and identify key software functionalities.
    *   **Decompose the system** into a list of testable functions (functional requirements).
    *   **Identify non-functional requirements** (e.g., performance, security) that need special attention.
    *   **Prioritize test items** based on risk and business importance.

*   **Entry and Exit Criteria for Testing:**
    *   **Entry Criteria (Test Readiness):** Defines when testing can begin. This includes having a stable build, completed requirements documents, a ready test environment, and a clear team structure.
    *   **Exit Criteria:** Defines when testing is complete. This is not just about meeting a deadline but about achieving a predefined quality target (e.g., test pass rate, no outstanding critical defects).
