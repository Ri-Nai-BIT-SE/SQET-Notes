### 第一部分：第一小组中期测试报告 (汇报人：王绍汝)

该小组（自称“吴彦祖”小组）汇报的是一个为大学“电脑诊所”开发的后端预约系统。

**1. 测试概述:**
*   **测试范围:** 覆盖了代码静态质量、核心业务功能、系统安全性及 API 性能与规范性。
*   **项目进度:** 截至汇报时，已100%完成中期计划的既定任务。
*   **测试环境:** 后端项目基于 Python 语言。

**2. 主要测试发现与建议:**

*   **静态代码与配置问题:**
    *   **代码规范:** 命名风格混乱（驼峰与蛇形混用）、使用“魔术数字”（如列表索引定义状态）而非枚举、部分逻辑缺乏注释和异常处理。
    *   **安全配置:**
        *   **高危:** CORS 配置允许所有来源的跨站请求，易受 CSRF 攻击。
        *   **中危:** API 签名使用已被证实存在碰撞风险的 MD5 算法，建议升级至 SHA-256。
        *   **风险:** 敏感信息（如数据库密钥）硬编码在代码中，建议从环境变量读取。
*   **数据库设计问题:**
    *   **亮点:** 使用了联合约束，保证了数据的唯一性。
    *   **严重隐患:** 用户的设备开机密码在数据库中以**明文存储**，存在严重的隐私泄露风险。
    *   **设计缺陷:** 工单中冗余存储了用户姓名电话，违反数据库设计范式，可能导致数据不一致。
*   **核心业务功能测试:**
    *   **身份认证:** 用户创建、登录、凭证校验、权限隔离等功能流程顺畅，异常场景（如密码错误、无 Token 访问）处理符合预期。
    *   **权限边界:** 成功验证了普通用户无法访问管理员接口（纵向越权），学生 A 也无法修改学生 B 的数据（横向越权），数据隔离性良好。
    *   **核心流程:** 预约工单从提交到删除的整个生命周期状态流转正确，功能完整健壮。
*   **输入防御测试:**
    *   注入 XSS 脚本、SQL 语句和超长文本均被系统成功防御，未出现异常。
*   **API可用性测试:**
    *   **问题:**
        1.  **语言不统一:** API 错误提示信息中英文混杂。
        2.  **响应格式不统一:** 错误响应的 JSON 结构存在 `detail`, `field_errors`, `message` 等多种格式，增加了前端处理难度。
        3.  **接口风格不一致:** 分页接口有的返回标准分页结构，有的直接返回数组。
    *   **优点:** 接口响应速度极快（10ms以内），用户体验流畅。

**3. 总结与后续计划:**
*   **高危问题汇总:** 识别出三个最主要的高危问题：R01（跨站请求伪造风险）、R02（API 缺乏时间戳校验，有重放攻击风险）、R03（用户密码明文存储）。
*   **后续建议:** 优先修复上述高危漏洞，然后统一接口标准、重构代码。

**4. 问答环节:**
*   同学提问项目背景和测试思路。汇报人解释项目是为校内“电脑诊所”服务，测试重点在于保证普通学生的使用流畅性和防止技术背景学生进行恶意攻击。

### 第二部分：第五小组中期报告分析 (汇报人：赵雨彤)

该小组汇报的是一个前端网页游戏的测试项目。

**1. 测试概述:**
*   **协作方式:** 使用 GitHub Pages 部署，GitHub 进行版本控制，腾讯文档和 PingCode 进行协作管理。
*   **测试策略:** 由于游戏已开发完成，采取了“二次开发”的测试策略，即：冒烟测试 -> 系统测试（黑盒+白盒） -> 回归/专项测试。

**2. 主要测试发现与建议:**

*   **冒烟与端到端测试:**
    *   **发现:** 游戏在 PC 端可正常运行，但在模拟的移动端上存在严重的 UI 问题，如按钮被遮挡、背景图拉伸。
*   **黑盒测试 (动态):**
    *   **方法:** 使用 Selenium 工具，结合等价类、边界值、决策表和因果图等方法设计用例。
    *   **发现:**
        *   基础功能（登录、注册、剧情交互）自动化测试通过。
        *   **高危安全漏洞:**
            *   **XSS 攻击:** 登录注册处未对用户输入进行过滤，导致恶意脚本可被注入并存储在 `localStorage` 中。
            *   **存档对象注入:** 存档功能存在漏洞，可通过输入恶意构造的数据破坏游戏存档，导致程序崩溃。
*   **白盒测试 (静态与动态):**
    *   **静态:** 使用 ESLint 等工具扫描源码，发现代码逻辑虽简单，但存在高危安全隐患和大量不规范写法。
    *   **动态:** 设定了语句、分支、函数的覆盖率目标。简单模块覆盖率高，但复杂功能（如存档）的覆盖率和成功率均不理想。
*   **非功能性测试:**
    *   **兼容性:** 核心功能在多浏览器上可运行，但存在布局和样式问题。移动端兼容性问题严重。
    *   **可靠性:** 游戏长时间运行（30分钟，点击上千次）未崩溃。但断网重连后，背景图片无法加载。
    *   **性能:** 响应速度非常快，性能表现出色。
*   **用户体验测试:**
    *   检查了按钮交互、系统提示及 UI 风格的一致性。

**3. 总结与后续规划:**
*   **缺陷总结:** 共记录了 15 个急需解决的严重缺陷，主要集中在安全漏洞和移动端兼容性。
*   **下一步:** 优先修复已发现的缺陷，继续执行复杂的测试用例，并计划在未来开发新功能时采用更规范的“单元-集成-系统”测试流程。

### 第三部分：师生交流与教师点评

*   **同学互评:** 两位汇报人互相评价，认为对方在项目介绍的完整性、报告的系统性等方面各有优劣。
*   **教师总结与指导:** 老师对本次中期汇报进行了深入点评，并对所有学生提出了要求：
    *   **中期报告的目的:** 强调中期汇报是承上启下的关键环节，用于同步进度、发现偏差、调整计划，无论是学术研究还是职场工作都至关重要。
    *   **演示结构:** 建议采用“总-分-总”的结构。先给出结论性概述（计划完成度、有无偏离），再展开细节，最后总结。直接切入细节会让听众（评委）难以把握重点。
    *   **内容呈现:**
        *   应突出工作量和关键成果，如测试用例数量、通过率、缺陷等级分布等。
        *   建议多使用图表（饼图、柱状图、折线图）来直观地展示数据，而非单纯罗列。
    *   **专业性:** 指出在正式报告中应避免使用“玩梗”或不规范的名称（如“吴彦祖”小组），PPT 风格也应保持专业、简洁。
    *   **答辩技巧:** 提醒学生在 final presentation 时，需要在有限的时间内（如5分钟）清晰地展示项目核心和测试成果，要充分考虑评委的接受习惯。
